Goal

Developed a scalable fault-tolerant Multi-Paxos based distributed transaction processing system supporting money transfers and account balance reads. Data was sharded between multiple clusters and cross shard transactions were handled with the 2 Phase Commit protocol.

Process

Each node lives on its own seperate process and communicates with other nodes using gRPC. The system is designed to handle node failures gracefully using the Multi-Paxos consensus algorithm to ensure consistency across replicas. Transactions are processed asynchronously to maximize throughput, and a benchmarking suite was developed to evaluate system performance under various workloads.

Highlights

Was able to process 3000 transactions per second on a cluster of 5 nodes. The benchmarking system allows users to generate workloads with different cross-shard transaction percentages, read/write ratios and workload distributions. Additionally hypergraph partitioning was implemented to re-shard data members to minimize cross-shard transactions.

Overview

The application is a distributed transaction processing system designed to handle money transfers and account balance reads across multiple nodes. Each transaction, such as transferring funds between accounts or querying balances, is processed reliably and consistently, even in the presence of node failures. gRPC is used for efficient inter-node communication, enabling fast and scalable message passing between distributed components. The system is engineered for high performance, supporting thousands of transactions per second through asynchronous processing and optimized data sharding.


Data Sharding and Coordination

Data sharding is used to partition account data across several clusters, allowing the system to scale horizontally. Each shard manages a subset of accounts, and transactions involving accounts from different shards require coordination. To ensure atomicity and consistency for these cross-shard transactions, the system implements the 2 Phase Commit (2PC) protocol. This protocol coordinates between shards to guarantee that either all parts of a transaction succeed or none do, preventing partial updates.


Resharding

To further optimize performance, hypergraph partitioning is employed for data sharding. This technique intelligently assigns accounts to shards based on transaction patterns, minimizing the number of cross-shard transactions and improving overall throughput.


Benchmarking and Performance Testing

The benchmarking suite enables users to simulate various workloads, specifying the ratio of read to write operations and controlling the distribution of transactions. For example, workloads can be centered around specific accounts or regions, or can vary the percentage of cross-shard transactions. This flexibility helps evaluate system performance under realistic and diverse scenarios.
